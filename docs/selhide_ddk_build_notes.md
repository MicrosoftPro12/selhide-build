# selhide DDK Build Notes

## Scripts

- `./build_selhide_ddk.sh`: run inside a DDK container, or any environment with
  a usable Android kernel build tree.
- `./run_selhide_ddk_container.sh`: optional launcher for
  `ghcr.io/ylarod/ddk-min:<kmi>-20260313` using docker, podman, or udocker.

## In An Existing DDK Container

```sh
cd /workdir
KMI=android16-6.12 ./build_selhide_ddk.sh
```

Output defaults to the current directory:

```text
./selhide-android16-6.12.ko
```

Useful overrides:

```sh
SELHIDE_SRC=/workdir/selhide-popsicle \
OUTDIR=/workdir/out-ddk \
OUT_NAME=selhide-test.ko \
KMI=android16-6.12 \
./build_selhide_ddk.sh
```

If the container has no `ddk` command but has a prepared kernel build tree:

```sh
KDIR=/path/to/kernel/build FORCE_MAKE=1 ./build_selhide_ddk.sh
```

## Launch A New Container

On a host with docker or podman:

```sh
KMI=android16-6.12 ./run_selhide_ddk_container.sh
```

For fuxi-style Android 13/14 5.15 work:

```sh
KMI=android13-5.15 ./run_selhide_ddk_container.sh
```

For udocker:

```sh
ENGINE=udocker KMI=android16-6.12 ./run_selhide_ddk_container.sh
```

The current `ghcr.io/ylarod/ddk-min:*` images checked here are `linux/amd64`
single-architecture images. On Xiaomi/Android Termux (`linux/arm64`), udocker
therefore needs x86_64 user-mode emulation. Without `qemu-x86_64` or
`qemu-x86_64-static`, the container may print:

```text
Warning: qemu required but not available
proot info: vpid 1: terminated with signal 4
```

That is not a selhide compile failure; the DDK container never reached
`./build_selhide_ddk.sh`. Use one of these paths instead:

- Run `./run_selhide_ddk_container.sh` on an x86_64 Linux host with docker or
  podman.
- Install usable x86_64 qemu-user support for udocker, then retry. This will be
  slow and may still be fragile under nested proot.
- For local smoke testing only, use the checked-out ACK tree:

```sh
OUTDIR=/workdir/out-ddk-test \
OUT_NAME=selhide-local-ack.ko \
FORCE_MAKE=1 \
./build_selhide_ddk.sh
```

## What Has Been Verified Here

The local non-DDK fallback path works:

```sh
OUTDIR=/workdir/out-ddk-test OUT_NAME=selhide-script-test.ko FORCE_MAKE=1 ./build_selhide_ddk.sh
```

It produced a `p0.9-cleanaccess` `.ko` and `kallsyms_init_module --dry-run`
resolved all undefined symbols.

The true DDK path still needs to be run inside the DDK container. If it fails,
keep the full output; the likely failure modes are:

- DDK command expects a different external-module layout.
- DDK headers do not expose SELinux private headers in the same location.
- KCFI generated by DDK makes our current assembly KCFI wrapper redundant or
  conflicting.
- `CFLAGS_REMOVE_selhide_patch_memory.o` names differ in the DDK build.

## Important

This script only verifies whether the current popsicle source can compile in
DDK. It does not make the resulting `.ko` safe for other kernel profiles. fuxi
still needs a separate pre-6.6 source path before DDK compile is meaningful.
